# Schedule Intake

## When to use

사용자가 회의록, 메신저 대화, 이메일 초안, 업무 메모에서 내가 챙겨야 할 일정 후보를 찾아달라고 요청할 때 사용한다.

## Inputs

- 현재 태스크 폴더의 `raw-text.md`를 읽는다.
- 원문에 기준 시각이 있으면 상대 날짜를 해석할 때 사용한다.

## Output

- 결과는 `output/intake.md`에 쓴다.
- 아래 `Output template` 구조를 사용한다.

## Categories

아래 카테고리만 사용한다.

| Category | Use for |
|---|---|
| `meeting` | 회의, 콜, 면담, 세션 |
| `deadline` | 특정 날짜나 시간까지 끝내야 하는 일 |
| `follow_up` | 회신, 전달, 공유, 확인 |
| `preparation` | 다른 일정 전에 필요한 준비나 검토 |
| `personal` | 업무 텍스트에 섞인 개인 일정 |

## Personal relevance rules

다음 경우에는 사용자와 관련된 일정 후보로 본다.

- 사용자가 직접 이름으로 언급되거나 지목된 경우
- 사용자가 직접 하겠다, 확인하겠다, 보내겠다, 검토하겠다고 말한 경우
- 사용자 역할이 `나`로 표시된 메모나 할 일 목록에 포함된 경우
- 사용자가 가진 자료나 역할 때문에 다음 행동이 사용자에게 이어지는 경우

## Exclusion rules

다음 경우에는 일정 후보에서 제외한다.

- 다른 사람에게만 명확히 배정된 일
- 행동이나 시간이 없는 단순 배경 정보
- 캘린더에 넣을 만한 행동, 마감, 리마인더가 없는 내용
- 이미 끝났고 후속 조치가 필요 없는 내용

## Date and ambiguity handling

- `raw-text.md`의 기준 시각을 사용해 상대 날짜를 해석한다.
- 해석 가능한 경우에도 원문의 날짜/시간 표현을 근거로 보존한다.
- `다음주 이 시간`은 기준 시각이 있을 때 다음 주 같은 요일, 같은 시각으로 해석한다.
- `월요일 10시`처럼 요일과 시간이 함께 있으면 기준 시각 이후 가장 가까운 해당 요일과 시간으로 해석한다.
- `같은 시간`은 같은 대화 안의 가장 가까운 관련 일정이 있으면 그 일정의 시간을 기준으로 해석한다. 기준이 불명확하면 확인 필요로 남긴다.
- `후속 회의 전`은 정확한 시각이 아니라 의존 관계로 다룬다.
- `콜 전`, `회의 전`은 정확한 시각이 아니라 의존 관계로 다룬다.
- `오전 중`, `화요일 오전`, `회의 전`처럼 정확하지 않은 표현은 원문에 정확한 시각이 없으면 모호한 값으로 남긴다.
- 마감 시각과 잠정 실행 시간이 함께 나오면 둘 다 보존한다. 예를 들어 `다음주 이 시간 전까지`는 마감으로, `화요일 오전으로 생각`은 확인이 필요한 잠정 실행 시간으로 남긴다.
- 모호한 날짜나 시간은 임의로 확정하지 않는다.
- 부족하거나 모호한 값은 `확인 필요`에 적는다.

## Procedure

1. `raw-text.md`를 읽는다.
2. 기준 시각과 사용자 역할을 확인한다.
3. 사용자와 관련된 일정 후보를 찾는다.
4. 사용자 관련성 규칙과 제외 규칙을 적용한다.
5. 기존 카테고리 중 하나를 선택한다.
6. 짧은 원문 근거를 보존한다.
7. 기준 시각으로 안전하게 해석 가능한 상대 날짜만 해석한다.
8. 모호한 날짜나 시간은 `확인 필요`에 남긴다.
9. 사용자와 무관한 항목은 제외한 항목에 이유와 함께 적는다.
10. `output/intake.md`를 작성한다.

## Output template

`output/intake.md`는 이 구조로 작성한다.

```markdown
# 일정 후보 추출 결과

## 기준 정보

- 기준 시각:
- 입력 출처:
- 사용자 역할:

## 일정 후보

| id | 제목 | 카테고리 | 원문 근거 | 날짜/시간 표현 | 확인 필요 |
|---|---|---|---|---|---|

## 제외한 항목

| 원문 | 제외 이유 |
|---|---|
```
